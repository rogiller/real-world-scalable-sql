Where does the story begin?

Let's talk about Flex4 a bit.

  • It's a legacy Java single tenant app.
  • uses Spring/Hibernate/Jetty/MySQL
  • started in 2003'ish
  • It was built super rapidly.
  • Speed was never much of a consideration
  • little thought about structural stuff
  • Large methods & classes (13,000+ line class!)
It has a LOT of issues. We will focus on it's speed problem.

I mentioned it's built with HIBERNATE.

Hibernate is the problem right?! Just don't use it!
Hibernate really isn't the core issue. **It's what YOU ask it to DO** and unawareness of it's MO is the killer.
Yes, you could argue it's over-engineered. **But not here to debate that.**

Blaming Hibernate is like blaming the poor delivery person...

Hibernate is just doing what you asked. **It's up to you to ask it to do reasonable things.**
So what is the core issue?

It's your webby object graph.

What are the key graph woes?
- N+1 selects (lazy loaded collections**)
- Impossible Hibernate Collection bugz - Database IO counts through the roof
- Database timeout/locks from high load - Massive memory spikes (stack objects)
What questions do you have about all that context?
How did we try to fix it?
Hibernate 2nd level cache > it's an "across session cache"
This might have helped.... **my gut is it hurt more than it helped.** > Areas of high churn create cache misses, leading to even more N+1.
Hibernate Collection Batching
The most successful performance tweak. > Lowers N+1 selects by loading multiple collections in one database trip.
Seriously, collection batching might be the best tip you take home. See [https://docs.jboss.org/hibernate/...](https://docs.jboss.org/hibernate/orm/5.3/userguide/html_single/Hibernate_User_Guide.html#fetching-batch).
We bypassed Hibernate by hand coding SQL... > But then Hibernate Second Level cache starts to bite you.
We built Ad Hoc Caches in hotspot areas. > But then you get cache eviction issues.
We tried to trim down object graph. > Mostly failed. Ripples out to far into the codebase.
What questions do you have about taming Hibernate?
What was the solution?
After much hand wringing... **We decided the Flex4 backend was categorically unscalable.**
A new backend was needed. We started a new Spring Boot/Data/Security backend... **on top of the existing database.** > this new backend is known as "Flex5"
Our rewrite approach is the "strangler pattern". Read [StranglerApplication](https://www.martinfowler.com/bliki/StranglerApplication.html) for more info.
Flex5 is live now and works in parallel with Flex4. > How we got the two systems to play nice is a story for another day. Spoiler: Just Google "Hazelcast".
How did we architect the data layer?
We knew Hibernate needed constrained. > We started with lighter entities having ManyToOne relations pointing to lightweight "identity" classes to avoid "web out". Look at `SoundTheme.groovy`.
> Eventually scrapped that and went with a straight up "row to object" model.
No @ManyToOne, @OneToMany, no collections. Only @Entity, @Table, @Id, @Column.
Flex5 Entity Example ```groovy @Entity @Table(name = Department.TABLE_NAME) class Department extends NameAuditEntity implements BusinessModel { public static final String TABLE_NAME = "st_biz_department" static class COLUMNS { public static final String DEPT_NAME = "dept_name" public static final String DEPT_CODE = "dept_code" public static final String ABBREVIATION = "abbreviation" public static final String ACCOUNTING_NAME = "accounting_name" } @Column(name = COLUMNS.DEPT_NAME, length = 128) String name @Column(name = COLUMNS.DEPT_CODE, length = 32) String code @Column(name = COLUMNS.ABBREVIATION, length = 16) String abbreviation @Column(name = COLUMNS.ACCOUNTING_NAME, length = 128) String accountingName } ```
What data layer tools did we end up using? Let's first look at tools we didn't pick...
Spring JdbcTemplate See [baeldung.com/spring-jdbc-jdbctemplate](https://www.baeldung.com/spring-jdbc-jdbctemplate)
```groovy List<Permission> findPermissionsForUser(String userId){ return jdbcTemplate.query('SELECT permission AS name, denied AS denied FROM rh_user_permissions WHERE userprofileid = ?', new RowMapper<Permission>() { @Override Permission mapRow(ResultSet rs, int rowNum) throws SQLException { return new Permission(name: rs.getString('name'), denied: rs.getBoolean("denied")) } }, userId) } ```
Ewww. **You should NOT have to manually map your DB results.**
sql2o Example [sql2o.org](https://www.sql2o.org/)
```java public List<Customer> fetchCustomers(customerId) { try (Connection con = sql2o.open()) { final String query = "SELECT id, name, address " + "FROM customers WHERE id = :customerId"; return con.createQuery(query) .addParameter("customerId", customerId) .executeAndFetch(Customer.class); } } ```
I like the "executeAndFetch" into class, but... Still too much boilerplate. > You should NOT need to worry about a connection or even have method body.
JDBI Example [jdbi.org](http://jdbi.org/)
```java @SqlUpdate("INSERT INTO user(id, name) VALUES (:id, :name)") void insertNamed(@Bind("id") int id, @Bind("name") String name); @SqlUpdate("INSERT INTO user(id, name) VALUES (:id, :name)") void insertBean(@BindBean User user); @SqlQuery("SELECT * FROM user ORDER BY name") @RegisterBeanMapper(User.class) List<User> listUsers(); ```
Getting much closer but still a little too much boilerplate. ```java @RegisterBeanMapper(User.class) //this makes me cringe a bit. ```
So what makes a great data layer tool? The absolute minimal boilerplate possible IMO.
**It should allow you to focus on your query.** > aka writing SQL shouldn't feel like sandpaper to your fingers. **It should be fun!**
What tools did we pick?
Spring Data JPA [baeldung.com/the-persistence-layer-with-spring-data-jpa](https://www.baeldung.com/the-persistence-layer-with-spring-data-jpa)
Spring Data was our starter data layer.
It has some really cool abstractions. Really good at getting records to/from database.
Spring Data Query Examples ```groovy @Query("SELECT o FROM SalesTaxRuleIdentity AS o ORDER BY o.name ASC") List<SalesTaxRuleIdentity> findAllIdentities() @Query('''SELECT icon FROM IconNode icon WHERE icon.name LIKE CONCAT('%',?1,'%') ORDER BY icon.name ASC''') Page<IconNode> searchIconNodes(String searchText, Pageable pageable) @Query(''' SELECT CASE WHEN (count(p.documentNumber) > 0) THEN TRUE ELSE FALSE END AS document_number_exists FROM ProjectElement p WHERE p.documentNumber = ?1 ''') boolean documentNumberExists(String documentNumber) ``` Note: What I like? - Zero boilerplate - Your entity object is populated and the data types figured out - You think in terms of classes & properties (vs database columns) - Query validated on startup == less integration tests! What I don't like? - Hibernate under the cover. Feels icky. - Session Cache gets stale with non-JPA DB updates - Explicit flush to been by non-JPA - Generated SQL output is a bit ugly - 1 degree removed from SQL. (Normally fine, until it isn't)
JOOQ [jooq.org](https://jooq.org)
Spring Data has limits in terms of dynamic query building. We found JOOQ to fit that need really well.
It has a builder pattern that is really powerful. However, a bit non-awesome to read. > Spring Data is our default GOTO until the problem feels like JOOQ land.
JOOQ Query Example ```java List<InventoryGroup> findByParentGroupId(String parentId) { return jooq.select(ID, GROUP_NAME, FULL_DISPLAY_STRING, ICON_ID, MANAGEMENT_GROUP_ID) .from(TABLE) .where(equalToOrIsNull(PARENT_GROUP_ID, parentId)) .orderBy(ORDINAL, GROUP_NAME) .fetchInto(InventoryGroup) } ``` Note: What I like? - Powerful builder pattern - Perfect for dynamic queries - Fantastic re-usability - Solid POJO mapping abilities (can interpret @Column JPA annotations) - Type-safe SQL queries What I don't like? - Hard to read large queries - Requires a method body - Not new dev friendly - It's "generate POJO" feature. Feels icky. We don't use.
Is it working?
YES, it's been a huge success for Flex5 speed.
Things we still do...
We still have to watch out for database round trips.
Since we only have ID's references... We try to bag them up and fetch all at once, etc.
We use a SPARSE select when pulling big classes. We have a bunch of tables that have 100's of columns.
Sparse SELECT example ```groovy @Query(''' SELECT eqDef.id as id, eqDef.prepManifestTypeId as prepManifestTypeId, eqDef.prepAutoScanMode as prepAutoScanMode, eqDef.prepAutoScanMatchMode as prepAutoScanMatchMode, eqDef.autoPopulateNumberEnabled as autoPopulateNumberEnabled, eqDef.numberingSchemeId as numberingSchemeId, eqDef.numbered as numbered FROM EquipmentListDefinition AS eqDef WHERE eqDef.id = ?1 ''') Map findSparseByIdForPrepProcess(String definitionId) ```
> Caching is a smell if it's your first instinct.
We use recursive MySQL stored procedures. They help us locate id sets for parent/child relationships.
And finally, how does our data layer translate to JSON?
The beauty of this approach is that we have no domain/DTO translation layer.
The data comes out lean from the database. > and our entities serialize directly across the wire.
