Where does this story begin?
Let's talk about Flex4
Legacy Spring/Hibernate/MySQL application started in 2003'ish Built rapidly w/o a lot of thought about maintainability
Large methods & classes (13,000+ line classes!) It's worse part is it's immense object graph.
For those unfamiliar with object graphs, let me show you some Flex4 code.
ProjectElement, ProjectElementLineItem
What did this object graph lead to?
Gigantic SQL selects (think 50MB for a single Quote line update). N+1 selects (often from lazy loaded collections).
Impossible Hibernate Collection bugz. Database IO counts through the roof.
Database timeout/locks from high load. Massive memory usage due to large amounts of data in memory per request.
How did we try to fix it?
Hibernate 2nd level cache
This might have helped, but my gut is that it ended up hurting more than it helped. In areas of high churn, you can get a lot of cache misses, leading to even worse N+1 situations.
Hibernate Collection Batching
This was probably the most successful performance tweak. It helps avoid N+1 selects by loading multiple collections in one database trip.
Native SQL
Often we just gave up on Hibernate and hand coded SQL.
Ad Hoc Caches in hotspot areas.
Tried to trim down object graph. Mostly failed. Ripples out to far.
What was the solution?
After much debate, we realized the Flex4 backend was fundamentally unscalable.
New Backend Was Needed We started a new Spring Boot/Data/Security backend on top of then existing database.
It now works in paralel with the Flex4 backend. How we got the two systems to play nice together is a story for another day. Spoiler: Just Google "Hazelcast".
How did we architect the data layer?
We knew Hibernate needed constrained. Started with lighter entities with relations pointing to an "identity" class to avoid "web out".
Eventually screwed that and went with straight up "row to object" model.
No @ManyToOne, @OneToMany, no collections, nothing. Just straight up @Column, period.
Flex5 Entity Example ```groovy @Entity @Table(name = Department.TABLE_NAME) class Department extends NameAuditEntity implements BusinessModel { public static final String TABLE_NAME = "st_biz_department" static class COLUMNS { public static final String DEPT_NAME = "dept_name" public static final String DEPT_CODE = "dept_code" public static final String ABBREVIATION = "abbreviation" public static final String ACCOUNTING_NAME = "accounting_name" } @Column(name = COLUMNS.DEPT_NAME, length = 128) String name @Column(name = COLUMNS.DEPT_CODE, length = 32) String code @Column(name = COLUMNS.ABBREVIATION, length = 16) String abbreviation @Column(name = COLUMNS.ACCOUNTING_NAME, length = 128) String accountingName } ```
Basically nobody does data access right.
sql2o Example
```java public List<Customer> etchCustomers(customerId) { try (Connection con = sql2o.open()) { final String query = "SELECT id, name, address " + "FROM customers WHERE id = :customerId"; return con.createQuery(query) .addParameter("customerId", customerId) .executeAndFetch(Customer.class); } } ```
Way way to much boilerplate. You should NOT need a method body for normal record to POJO. Period.
JdbcTemplate Example
```groovy List<Permission> findPermissionsForUser(String userId){ if(StringUtils.isBlank(userId)){ return Collections.emptyList() } userId = StringUtils.trim(userId) return jdbcTemplate.query('SELECT permission AS name, denied AS denied FROM rh_user_permissions WHERE userprofileid = ?', new RowMapper<Permission>() { @Override Permission mapRow(ResultSet rs, int rowNum) throws SQLException { return new Permission(name: rs.getString('name'), denied: rs.getBoolean("denied")) } }, userId) } ```
Yuck. Repeat after me. YOU SHOULD NOT HAVE TO MANUALLY MAP YOUR RESULTS. PERIOD.
JDBI Example
```java @SqlUpdate("INSERT INTO user(id, name) VALUES (:id, :name)") void insertNamed(@Bind("id") int id, @Bind("name") String name); @SqlUpdate("INSERT INTO user(id, name) VALUES (:id, :name)") void insertBean(@BindBean User user); @SqlQuery("SELECT * FROM user ORDER BY name") @RegisterBeanMapper(User.class) List<User> listUsers(); ```
Nope still to much boilerplate. `@RegisterBeanMapper(User.class)` makes me shiver.
So what makes a great data layer tool? The absolute minimal boilerplate possible. It should allow you to focus on your actual query and almost nothing else.
The tools we picked.
Spring Data
Spring Data was our starter data layer.
It has some really cool abstractions. Really good at getting records to/from database.
```groovy @Query("SELECT o FROM SalesTaxRuleIdentity AS o ORDER BY o.name ASC") List<SalesTaxRuleIdentity> findAllIdentities() @Query(''' SELECT ruleIdentity FROM FinancialDocumentDefinitionEnabledSalesTaxRule AS enabledRule INNER JOIN SalesTaxRuleIdentity AS ruleIdentity ON ruleIdentity.id = enabledRule.salesTaxRuleId WHERE enabledRule.definitionId = ?1 ORDER BY ruleIdentity.name ASC ''') List<SalesTaxRuleIdentity> findAllIdentitiesEnabledForDefinitionId(String enabledForDefinitionId) ```
What I like? - Zero boilerplate. - The target entity and it's corresponding property types are just figured out and hydrated. - The parameters are just mapped in by ordinal (you can do named ones). - You deal at the class level & property level. - It's declarative and frickin' readable. - The JPQL queries are validated at startup.
What I don't like? - It's still uses Hibernate under the cover. Meh. Scar tissue. - Session cache sucks when you have other data access tools in your app. - e.g. you insert with JOOQ, but JPA doesn't know. - You have to explicitly flush for it to go the database (before the transaction ends). - Generated SQL output is a bit ugly. - You are 1 degree removed from SQL. Which is normally fine, until it isn't. - You want to use a SQL feature JQL doesn't support.
JOOQ [JOOQ website](https://jooq.org)
Spring Data has limits in terms of criteria based query building. We found JOOQ and fell in love. Er, at least some of us did.
It has a builder pattern that is really powerful, if a bit non-awesome to read. Spring Data is our default GOTO until the problem feels like JOOQ land.
```groovy List<InventoryGroup> findByParentGroupId(String parentId) { return jooq.select(ID, GROUP_NAME, FULL_DISPLAY_STRING, ICON_ID, MANAGEMENT_GROUP_ID) .from(TABLE) .where(equalToOrIsNull(PARENT_GROUP_ID, parentId)) .orderBy(ORDINAL, GROUP_NAME) .fetchInto(InventoryGroup) } ```
What I like? - Powerful builder pattern. - Perfect for dynamic queries. - Fantastic reusability. - Solid POJO mapping abilities (can interpret @Column JPA annotations). - Type-safe SQL queries.
What I don't like? - Hard to read large queries. aka more difficult to maintain. - Requires a method body. - Probably hard for a junior or new to JOOQ dev to grok. - It wants you to use it's "generate jazz" feature which feels really messy. - We didn't. We created our own fields. Which takes up some time.

Is it working?
By and large, it's been a huge success in terms of speed. We still have to watch out for database round trips.
We'll normally bag up some ID's and fetch all at once, etc.
We use a SPARSE select often. We have a bunch of tables that have 100's of columns.
We'll do a sparse "use case select". e.g. "findSparseByIdForPrepProcess"
We use recursive database functions. They help us locate id sets for parent/child relation ships.
How does the our data layer translate to JSON?
The beauty of this approach is that we have no domain/DTO translation layer.
The data comes out lean from the database. Our entities serialize directly out across the wire.
